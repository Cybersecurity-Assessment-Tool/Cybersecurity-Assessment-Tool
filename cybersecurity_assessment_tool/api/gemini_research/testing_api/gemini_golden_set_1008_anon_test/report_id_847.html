Risk Assessment

Identified Risks
Risk Name Overview Severity
Lack of MFA for Email The absence of Multi-Factor Authentication (MFA) for email access leaves accounts highly vulnerable to compromise through phishing, brute-force, or credential stuffing attacks. A compromised email account can lead to Business Email Compromise (BEC), sensitive data exposure, and serve as a foothold for further network intrusion. Critical
Missing Acceptable Use Policy (AUP) Without an established Acceptable Use Policy, employees may not be fully aware of the appropriate and inappropriate use of organizational IT assets, data, and networks. High

High

Recommendations

To mitigate the identified risks and significantly enhance the cybersecurity posture of [Organization Name], the following actionable recommendations are provided.

Immediate Actions

  • Implement Multi-Factor Authentication (MFA) for Email Accounts:

    • Enforce MFA for all user accounts accessing organizational email. This is a foundational control to protect against credential compromise.

    • Provide clear instructions and support to employees for MFA enrollment and usage.

  • Develop and Implement an Employee Acceptable Use Policy (AUP):

    • Create a comprehensive AUP that clearly defines the acceptable and unacceptable use of all company-provided IT resources, including computers, networks, internet access, email, and data.

    • Ensure all employees read, understand, and formally acknowledge the AUP upon hiring and periodically thereafter.

  • Mandatory Security Awareness Training for New Employees:

    • Establish and integrate mandatory security awareness training as a critical component of the onboarding process for all new hires.

    • The training should cover common cyber threats (e.g., phishing, malware), company security policies, reporting procedures for incidents, and the importance of secure behavior.

Technical Controls

  • Secure SSH Configuration for Port 22:

    • Strong Authentication: Enforce key-based authentication (e.g., SSH keys) and disable password-based authentication if feasible. If passwords are necessary, require strong, complex passwords.

    • Disable Root Login: Configure the SSH server to disallow direct root login.

    • Restrict Access: Implement IP-based firewall rules to limit SSH access only to specific trusted source IP addresses or networks.

    • Non-Standard Port: Consider moving the SSH service to a non-standard port to reduce automated scanning and attack attempts (though this does not provide security by obscurity).

    • Regular Patching: Ensure the SSH server software is consistently patched and updated to the latest stable version to address known vulnerabilities.

    • Rate Limiting/IDS/IPS: Implement intrusion detection/prevention systems (IDS/IPS) or firewall rules with rate limiting to block repeated failed login attempts and prevent brute-force attacks.

  • Conduct Regular Vulnerability Scanning:

    • Perform periodic, authenticated vulnerability scans of all network-facing systems. This will provide detailed information on service versions, configurations, and known CVEs (Common Vulnerabilities and Exposures), enabling more precise risk assessment and targeted remediation.
